JPA ([[Java - Index|Java]] Persistence API или пакет `jakarta.persistence`) - спецификация (набор правил, интерфейсов и аннотаций), которая описывает, как Java-объекты должны сохраняться в [[СУБД - INDEX|реляционную базу данных]] и подгружаться из нее.

Основная задача JPA - реализовать концепцию [[ORM - Object-Relational Mapping|ORM]]. Она решает проблему несоответствия между классами в Java и табличной структурой [[SQL - Index|SQL]].

---
## 1. Объявление сущности 

- **`@Entity`**: объявление класса как сущности JPA
- **`@Table`**: необязательный параметр взаимодействия сущности со структурой базы данных. Все параметры опциональны:
	- `name`: имя таблицы в базе данных для сущности
	- `schema`: указание конкретной схемы в некоторых СУБД (напр. [[Postgres - Index|Postgres]]) для таблицы
	- `catalog`: указание конкретной базы данных в некоторых СУБД (напр. [[MySQL - Index|MySQL]]) внутри экземпляра сервера
	- `uniqueConstraints`: автоматически создаваемое в БД ограничение уникальности (составной ключ таблицы), затрагивающее несколько столбцов
		- `@UniqueConstraint` используется для указания конкретного экземпляра
	- `indexes`: автоматическое создание индексов таблицы в БД
		- `@Index` используется для указания конкретного экземпляра
- **`@Embeddable`**: помечает класс как компонент, который потом может использоваться как часть класса с аннотацией `@Entity` без своего жизненного цикла. Не создает отдельную таблицу в БД, а дополняет уже существующую
- **`@MappedSuperclass`**: // TODO
- **`@SecondaryTable`**: необязательный параметр для указания вторичной таблицы. Используется, когда нужно привязать дополнительную таблицу к основной. `@SecondaryTables` используется для указания массива таблиц
> [!BUG] При каждом запросе создает JOIN запрос даже если данные не используются, лучше заменить связью @OneToOne c [[ORM - Object-Relational Mapping|LAZY]] инициализацией**

**Пример использования**:
```java
@Entity
@Table( 
	name = "class_table", 
	schema = "my_schema",
	uniqueConstraints = { 
		@UniqueConstraint( 
			name = "ek_email_username", 
			columnNames = {"email", "name"} 
		) 
	},
	indexes = { 
		@Index(name = "idx_name", columnList = "name")
)
public class User {
	private Long id;
	private String name;
	private String email;
	public User() {}
}
```

## 2. Первичный ключ

- **`@Id`**: помечает поле как первичный ключ
- `@GeneratedValue:` стратегия, описывающая поведение базы данных при генерации ID, через параметр `strategy =`:
	- `GenerationType.IDENTITY`: базовое инкрементирование ID внутри БД
	- `GenerationType.SEQUENCE`: создание специальной сущности БД  - Sequence, из которой JPA может сразу зарезервировать батч айдишников 
	- `GenerationType.UUID`: генерация UUID в качестве ID сущности
	- `GenerationType.TABLE`: создание отдельной служебной таблицы, в которой хранится текущее значение последнего ID. При каждом сохранении сущности JPA блокирует строку, обновляет значение и использует его
	- `GenerationType.AUTO`: JPA сама выбирает стратегию на основе используемой СУБД

> [!IMPORTANT] Вывод: из вышеперечисленных стратегий лучшими будут SEQUENCE и UUID. IDENTITY может вызвать проблемы с производительностью при пакетных вставках, TABLE из-за блокировок является самым медленным способом, а AUTO может быть непредсказуемой - очень большой риск получить общую SEQUENCE для всех таблиц

**Пример использования**:
```java
@Entity
public class User {
	@Id
	@GeneratedValue(strategy = GenerationType.SEQUENCE)
	private Long id;
}
```

## 3. Аннотации для маппинга полей

- **`@Column`**: необязательный параметр для объявления колонки в таблице. Все параметры опциональны:
	- `name`: задает имя таблице (без указания JPA может некорректно обработать camelCase в snake_case)
	- `unique`: задает `uniqueConstraint` из одного поля (уникальность значения) (def. false)
	- `nullable`: задает `nullable` колонке (def. true)
	- `insertable` и `updatable`: указывает, можно ли вставить или изменить поле (def. true). Полезно для полей типа `created_at`
	- `columnDefinition`: позволяет написать SQL-фрагмент, который будет использоваться при создании колонки
	- `options`: позволяет написать SQL-фрагмент, который дополнит создание колонки - не работает совместно с `columnDefinition`
	- `table`: указание, из какой таблицы брать поле (при использовании `@SecondaryTable`)
	- `length`: длина строки ([[SQL - Типы данных|varchar или varbinary]])
	- `precision` и `scale`: работа с дробными числами ([[SQL - Типы данных|decimal или numeric]])
		- `precision`: общее количество цифр в числе
		- `scale`: количество цифр после запятой
	- `secondPrecision`: точность дробной части секунды
	- `check`: задает SQL-ограничение `CHECK` в аннотации
	- `comment`: добавляет текстовый комментарий к колонке в базе данных
- **`@Enumerated`**: работа с ENUM. Параметры:
	- `@EnumType.ORDINAL`: (def.) сохраняет ENUM элементы в виде числового индекса
	- `@EnumType.STRING`: сохраняет ENUM элементы в виде текста ()
> [!IMPORTANT] Рекомендуется использовать STRING для читаемости значений в базе данных
- **`@Version`**: объявление version-поля сущности. Используется для защиты от одновременного редактирования строки
- **`@Transient`**: служебное поле, которое не должно сохраняться в БД
- **`@Embedded`**: использует поле из `@Embeddable` сущности
- **`@AttributeOverride`**: переименовывает поле для конкретного экземпляра `@Embedded` или `@MappedSuperclass`. Параметры:
	- `name`: имя поля
	- `column`: указание новых параметров колонки через `@Column`
- **`@Lob`**: Large Objects, если нужно сохранить огромный текст или бинарный файл в БД
	- String превратится в [[SQL - Типы данных|CLOB или TEXT]]
	- byte[] превратится в [[SQL - Типы данных|BLOB]]
- `@Temporal`: хранение времени в поле (используется для устаревшего `java.util.Date`). Указывает, что именно должна хранить колонка:
	- `TemporalType.DATE` - дата, `.TIME` - время, `.TIMESTAMP` - и то, и другое
