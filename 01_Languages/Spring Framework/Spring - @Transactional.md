Транзакции в [[Spring - Index]] - механизм управления целостностью данных в БД через [[Spring - AOP]]. Транзакция позволяет или полностью выполнить кусок кода, или не выполнить вовсе.

---
## 1. Как работает "под капотом"

1. При помечании метода аннотацией `@Transactional`, Spring создает прокси-объект вокруг бина
	- Когда другой класс вызывает этот метод, метод вызывается не напрямую, а через прокси-объект
2. Прокси обращается к [[Spring - Менеджеры ресурсов|TransactionManager]], который берет одно из [[Java - JDBC|JDBC-соединений]] из [[Пулы соединений|пула HikariCP]], и делает `connection.setAutoCommit(false)`, запрещая соединению что либо сохранять в БД автоматически
3. Дальше происходит обычное выполнение кода в методе, но все взаимодействия с БД (именно изменения данных) добавляются в Persistence Context, пока код не выполнится до конца
4. Когда выполнение метода завершается, происходит flush:
	- Spring/[[Spring - Hibernate|Hibernate]] синхронизирует состояние объектов из памяти с БД и отправляет запросы в БД
5. **Commit**: база применяет все запросы на изменения данных, соединение возвращается в пул, setAutoCommit ставится обратно на true

**Особенность самовызова:** когда один из методов класса, помеченный `@Transactional` вызывает внутри себя метод из этого же класса, тоже помеченный `@Transactional`, новая транзакция может сработать не так, как хочет программист.
- Вызов пройдет в обход прокси-объекта (никакие параметры на `@Transactional` второго метода не сработают)
- Второй метод неизбежно станет частью транзакции первого метода
- Также если второй метод будет вызван методом внутри одного класса, не имеющим `@Transactional`, транзакция просто не создастся
**Главным решением будет выносить такие методы в отдельный слой (класс)**

## 2. [[SQL - Изоляция транзакций#2. Уровни изоляции в SQL - Index SQL|Уровни изоляции]] (Isolation Levels)

Параметр **Isolation** позволяет управлять уровнем изоляции внутри транзакции.
По умолчанию в `@Transactional` используется `Isolation.DEFAULT`, значение которого исходит из [[СУБД - INDEX|СУБД]], которая используется в приложении.

Чтобы транзакция использовала определенный уровень изоляции, TransactionManager открывает новое соединение, и Spring выполняет команду в [[Java - JDBC|JDBC-соединении]] с установкой нужного уровня изоляции (напр. `SET TRANSACTION ISOLATION LEVEL ...`).
Только после этого код метода начинает выполняться. После завершения транзакции соединение приходит в изначальное состояние.

## 3. Стратегии распространения (Propagation)

Параметр **Propagation** позволяет управлять тем, как метод будет вести себя относительно уже существующей транзакции. Исходит из того, является ли вызов метода частью другой транзакции или нет.

1. **REQUIRED** (default): использует текущую транзакцию или создает новую, если ее нет
2. **REQUIRES_NEW**: приостанавливает текущую транзакцию и создает новую. Если вторая падает - первая продолжает работу (если обработает в try-catch)
3. **NESTED**: вложенная транзакция, которая создает точку отката. Если вторая упала - первая продолжает работу (если обработает в try-catch). Если первая падает - вторая тоже заканчивает работу
4. **SUPPORTS**: использует текущую транзакцию или выполняется без транзакции
5. **NOT_SUPPORTED**: приостанавливает текущую транзакцию, если она есть
6. **MANDATORY**: использует текущую транзакцию или кидает исключение
7. **NEVER**: кидает исключение если есть транзакция
