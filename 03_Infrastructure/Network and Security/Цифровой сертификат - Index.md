Цифровой сертификат (файл стандарта X.509) - электронный документ, связывающий публичный ключ сервера с личностью его владельца (чаще всего домена) и подтверждается цифровой подписью доверенного центра сертификации (CA).

---
## 1. Поля сертификата

- **Serial Number:** Уникальный идентификатор сертификата в базе данных конкретного CA.
- **Subject**: кому выдан сертификат (домен)
- **Public Key**: математическая часть пары ключей, используется для безопасного согласования общего секрета при установке соединения
- **Issuer**: Название CA, который подтвердил личность
- **Срок действия** по датам Not Before и Not After
- **Цифровая печать CA**: печать CA, созданная их приватным ключом

---
## 2. Центры сертификации и Chain of Trust

Центр сертификации (CA) - доверенная организация, которая проверяет владельцев сайтов и подтверждает их личность, выдавая цифровые сертификаты. 

Выдача сертификатов не ограничивается проверкой сертификата в одном CA. Существует корневой CA, который имеет приватный, максимально закрытый ото всех ключ, и публичные ключи которого вшиты в браузеры и ОС. Этим ключом подписаны сертификаты дочерних CA, которые уже выдают цифровые подписи серверам.

Основные функции:

- **Валидация**: перед выдачей сертификата CA проверяет владение доменом (например, просит разместить файл на сервере или добавить DNS запись)
- **Выпуск**: CA берет публичный ключ сервера, добавляет к нему данные о владельце и делает цифровую печать
- **Отзыв**: если приватный ключ сервера украден, то CA заносит сертификат в черный список по запросу владельца

---
## 3. Алгоритм проверки сертификатов

1. Когда пользователь заходит на сайт, сервер выдает ему **цепочку сертификатов**, состоящую из:
	1. **Листовой (Leaf) сертификат:** сертификат домена сайта
	2. **Промежуточный (Intermediate) сертификат**: сертификат CA, подтверждающий его право на выпуск сертификатов
2. Первичная проверка - Subject в сертификате совпадает с доменом, действителен ли срок действия
3. Проверка подписей цепочки сертификатов - криптография:
	1. Вычисляется хэш содержимого сертификата
	2. С помощью публичного ключа издателя расшифровывается цифровая подпись, прикрепленная к сертификату
	3. Сравниваются хэши содержимого сертификата и расшифровки цифровой подписи.
	4. Таким образом ключом из промежуточного сертификата проверяется подпись листового сертификата, ключом корневого сертификата внутри ОС или браузера проверяется подпись промежуточного сертификата
4. Проверка статуса отзыва по черным спискам (**Через CRL и OCSP**):
5. **Handshake** между сервером и клиентом

---
## 4. Алгоритмы черного списка

- **CRL (Certificate Revocation List)**: файл, публикуемый CA, в котором перечислены серийные номера отозванных сертификатов:
	- Браузер скачивает этот файл и сверяет серийные номера 
	- Минус: огромный размер файла и задержки обновлений
	
- **OCSP (Online Certificate Status Protocol)**: сетевой протокол для проверки статуса отзыва цифровых сертификатов в реальном времени:
	- Браузер отправляет запрос на специальный сервер CA (OCSP-ответчику) с серийным номером сертификата
	- Ответчик отвечает: good (валиден), revoked (отозван), unknown (статус неизвестен)
	- Минус: запрос замедляет загрузку сайта, информация о посещенных сайтах открывается третьей стороне
	
- **OCSP Stapling**: замена OCSP, решающая проблему приватности
	- Сервер раз в несколько часов обращается к CA и получает статус сертификата вместе с подписанной timestamp (временной меткой), гарантирующей, что ответ CA не был изменен сервером
	- Информация и метка от CA прикрепляется к сертификату, выдаваемому браузеру
	- Браузер проверяет подлинность временной метки

---
## 5. Файлы сертификатов

- **Открытые (текстовые)**: самый частый формат для использования в Docker и Linux
	- **`.crt` / `.cer` / `.pem`**: Сам сертификат (публичная часть)
	- **`.key`**: Приватный ключ
	- **`fullchain.pem`**: Пример цепочки сертификатов
	
- **Контейнеры (бинарные)**: контейнер сертификата, защищенный паролем. И сертификат, и приватный ключ вшиты в него
	- **`.p12` / `.pfx`**: Стандарт индустрии.
	- **`.jks`**: Java KeyStore, специфичный для [[Java - Index|Java]]-среды формат.
	- **Зачем нужен**: защита приватного ключа при доступе к файловой системе
